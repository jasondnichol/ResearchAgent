Part 1 — Trade Setup Generation

Three methods produce complete trade setups (entry, SL, TP). A fourth method (Technical Analysis) produces directional signals only — no entry/SL/TP levels. All methods run every 5 minutes via the alert worker across multiple timeframes.

Key file locations (all under C:\tradesavvy\backend\signals\):
signal_smc.py · signal_elliott.py · signal_priceaction.py · signal_technical.py
Orchestrator: C:\tradesavvy\backend\alert_worker.py

1A. Smart Money Concepts (SMC)

🏦
Smart Money Concepts
Timeframes: 1h · 4h · 1d  |  File: signal_smc.py

Core Structures Detected

StructureHow It's FoundKey Params
Swing Highs/Lows Rolling window comparison: high[i] == max(high[i-w : i+w+1]) Window: 1h→7, 4h→10, 1d→10
BOS (Break of Structure) Higher high in uptrend = bullish BOS, lower low in downtrend = bearish BOS Trend continuation signal
CHoCH (Change of Character) Lower high in uptrend → bearish CHoCH, higher low in downtrend → bullish CHoCH Trend reversal signal
Order Blocks Candle where next candle surges > 2× ATR(14) in opposite direction Bullish OB = last bearish candle before surge up; returns last 8
Fair Value Gaps 3-candle gap: candle[i+1].low > candle[i-1].high (bullish FVG) Returns last 8 FVGs
Liquidity Levels Clusters of equal highs/lows within 0.1% over a 20-bar lookahead Top 5 deduplicated levels

Signal Scoring

Accumulates bull_score vs bear_score:

ConditionPoints
Near bullish/bearish order block (within 3% of last 3 OBs)+33 per match (max 99)
Last BOS direction matches+33
Unfilled FVG nearby (last 3)+17 per match (max 51)
Last CHoCH direction matches+17

Signal gate: score ≥ 50 AND higher than opposing score → BUY or SELL. Otherwise HOLD.

Confidence: min(100, max(bull_score, bear_score))

Trade Setup Derivation

LevelHow DerivedNotes
ENTRY Nearest order block price matching signal direction BUY → most recent bullish OB's low; SELL → most recent bearish OB's high. This is a limit-order level, often away from current price.
STOP LOSS Fixed 3% from entry BUY: entry × 0.97 · SELL: entry × 1.03. Does NOT adapt to structure or volatility.
TAKE PROFIT Nearest opposing structure level above/below entry BUY: min of (bearish OB prices + liq levels above entry). SELL: max of (bullish OB prices + liq levels below entry). Single TP only.

Risk/Reward: |TP − entry| / |entry − SL|. Only calculated when entry, SL, and TP all exist.

Conviction

TierConditions
HIGHconfidence ≥ 67 AND R:R ≥ 2.0 AND trend matches signal direction
LOWAny non-HOLD signal with a complete trade setup that doesn't meet HIGH
STRONGUpgraded when confidence ≥ 75 (BUY → STRONG_BUY)
Audit flags for SMC:
  • Fixed 3% stop — Ignores ATR/volatility. A 3% stop on BTC ($2,800) vs ADA ($0.02) have wildly different structural significance. Consider ATR-based or structure-based SL.
  • Entry can be far from price — OB prices can be stale (old candles). The 20% quality gate catches extremes but a 15% distance is still borderline.
  • Single TP only — No partial profit-taking. If the opposing structure is very far, R:R can be inflated and unrealistic.
  • "Near OB" is within 3% — Quite generous. Price might not actually retrace to the OB.

1B. Elliott Wave

🌊
Elliott Wave
Timeframes: 1h · 4h · 1d  |  File: signal_elliott.py

Wave Detection Pipeline

1
Zigzag filter — Smooths price into swing pivots using min-move thresholds: 15m: 0.8%, 1h: 1.5%, 4h: 2.5%, 1d: 3.0%
2
Pattern labeling — Takes most recent 6–10 pivots, tries to fit a 5-wave impulse. Tests multiple starting offsets. Falls back to partial 4-wave pattern if 5-wave fails.
3
Rule validation — Checks 3 cardinal Elliott rules:
  • Wave 2 cannot retrace > 100% of Wave 1
  • Wave 3 cannot be the shortest impulse wave
  • Wave 4 cannot overlap Wave 1 territory
rules_valid = True only if all pass.
4
Fibonacci projections — Extension targets (1.0×, 1.272×, 1.618×, 2.0×, 2.618× of Wave 1 length) and correction targets (23.6%, 38.2%, 50%, 61.8%, 78.6% of full move).

Signal Decision

Impulse DirectionAt Wave 2 or 4At Wave 1 or 3At Wave ≥ 5
BullishBUYBUYSELL (reversal)
BearishSELLSELLBUY (reversal)

Confidence Scoring

Current WaveRules ValidRules InvalidCap
Wave 2 or 4 (corrective — best entry)5535Hard cap: 70
(Elliott is subjective)
Wave ≥ 5 (completed)5030
Wave 1 or 3 (impulse)4525

Trade Setup Derivation

LevelHow DerivedNotes
ENTRY Current market price (round(current_price, 2)) Always at-market. An "optimal entry" (wave pivot) is computed but stored separately — not used as the tracked entry.
STOP LOSS Nearest wave pivot on the wrong side of the trade BUY: highest wave price still below current. SELL: lowest wave price still above current. Labeled e.g. "Wave 2".
TP1 Fibonacci extension index [0] — approx 1.0× extension Sourced from wave extension targets first. Falls back to correction retracement targets (38.2%, 50%, 61.8%) if insufficient extensions. Each TP has a label like "Wave 5 — 1.618×".
TP2 Fibonacci extension index [2] — approx 1.618× extension
TP3 Fibonacci extension index [min(4, len-1)] — approx 2.618× extension

R:R calculation: Based on TP1 (most conservative): |TP1 − entry| / |entry − SL|. If R:R can't be computed, the entire trade setup is cleared (entry, SL, all TPs set to null).

Conviction

TierConditions (ALL must be true)
HIGHR:R ≥ 2.0 AND stop distance ≥ 1.5% of entry AND wave is 2 or 4 AND rules_valid AND confidence ≥ 50
LOWAny non-HOLD with complete trade setup that doesn't meet HIGH
STRONGUpgraded when confidence ≥ 55 AND rules_valid
Audit flags for Elliott:
  • Entry = current price — This is always at-market. The signal is essentially "enter now." The tracker records this immediately, so the "entry reachability" check in resolution logic will always pass (entry ≈ signal price). This is inconsistent with SMC/PA which give limit-order entries.
  • SL from wave pivot — If the nearest wave pivot is very close (e.g. 0.5%), the stop is tight and gets flagged by the 1.5% conviction gate. But it can also be very far (10%+), creating a huge risk per trade.
  • Wave labeling is heuristic — Tries multiple starting points and takes the first valid one. The same price data can produce different wave counts depending on which window it starts from. Confidence is capped at 70 to reflect this subjectivity.
  • TP selection from extension indices — Uses array indices [0, 2, 4] regardless of how far apart the levels are. If extensions are clustered, TP1/TP2/TP3 can be very close together.

1C. Price Action + Volume

📊
Price Action + Volume
Timeframes: 1h · 4h · 1d  |  File: signal_priceaction.py

Core Analysis Pipeline

1
Swing detection — Window: 1h: 5, 4h: 10, 1d: 20. Identifies swing highs and lows.
2
Trend via HH/HL/LH/LL — Last 4 swing highs + last 4 lows. Counts higher-highs + higher-lows vs lower-highs + lower-lows. Produces trend (UP/DOWN/RANGE) + strength (0–100%).
3
S/R clustering — Groups swing lows into support zones, swing highs into resistance zones. Cluster tolerance: 0.5%. Sorted by touch count (strongest first). Top 5 per side.
4
Volume analysis — Current volume vs 20-period average. Spike = ratio ≥ 2.0×.

Signal Conditions

BUY Signal — All must be true:
  1. Trend = UPTREND
  2. Nearest support exists below price
  3. Price within 3% of nearest support
  4. Computed R:R ≥ 2.0
SELL Signal — All must be true:
  1. Trend = DOWNTREND
  2. Nearest resistance exists above price
  3. Price within 3% of nearest resistance
  4. Computed R:R ≥ 2.0

Trade Setup Derivation

LevelHow DerivedNotes
ENTRY Nearest S/R cluster price BUY: nearest support level. SELL: nearest resistance level. These are limit-order entries (like SMC), often below/above current price.
STOP LOSS 1.5 × ATR(14) beyond entry BUY: support − 1.5 × ATR. SELL: resistance + 1.5 × ATR. ATR = mean(high−low) over 14 candles. Volatility-adaptive.
TAKE PROFIT Nearest opposing S/R level BUY: nearest resistance. SELL: nearest support. Fallback: entry × 1.05 or entry × 0.95 (flat 5%). Single TP only.

Confidence

confidence = min(80, 40 + trend_strength // 3 + (15 if volume_spike else 0))

Range: 40–80. Base 40, up to +33 from trend strength, +15 for volume spike.

Conviction

TierConditions
HIGHconfidence ≥ 60 AND R:R ≥ 2.0
LOWAny valid trade setup not meeting HIGH
STRONGUpgraded when confidence ≥ 65 AND volume_spike = True
Audit flags for Price Action:
  • R:R ≥ 2.0 gate — This is a hard filter, which means PA only fires when setups look structurally good. This is probably the most reliable of the 3 methods for setup quality.
  • 5% fallback TP — When there's no opposing S/R level, TP defaults to a flat 5% from entry. This is somewhat arbitrary and can inflate R:R (since SL is ATR-based and likely smaller than 5%).
  • S/R clustering at 0.5% — Tight tolerance. In a volatile market, meaningful S/R could be missed if levels don't cluster within 0.5%.
  • "Within 3% of S/R" gate — This means price must be very near support/resistance. Good for quality but means many valid setups are skipped if price is even slightly far from S/R.

Part 2 — Signal Accuracy Tracking

Signals are recorded into signal_tracker for accuracy measurement. The alert worker runs every 5 minutes: first resolves pending signals, then generates new ones.

2A. Recording Signals

Pipeline: Signal → Database

1
Skip HOLD signals — Only BUY/SELL get tracked.
2
Confidence gate: confidence ≥ 50 required (constant TRACKING_MIN_CONFIDENCE).
3
Classify signal — Upgrade to STRONG_BUY / STRONG_SELL if thresholds met. Must produce non-null classification.
4
Per-timeframe dedup — Checks for an identical (symbol + methodology + timeframe + signal) within the dedup window:
TimeframeDedup WindowRationale
15m900s (15 min)Prevents re-recording within the candle period
1h3,600s (1 hr)One signal per hour candle
4h14,400s (4 hrs)One signal per 4h candle
1d86,400s (24 hrs)One signal per day — was the main fix for 1d duplicating 14×/day
5
Price validation: price > 0.
6
Quality gates (non-technical methods only, when entry + SL exist):
GateConditionWhy
Entry = SL |entry − SL| < 0.00001 Nonsensical setup — zero risk distance
Stale entry |entry − price| / price > 20% Order block too far from market — stale structure
Bad R:R R:R < 1.0 (when TP exists) Not a tradeable setup — more risk than reward

Technical signals are exempt from all quality gates (they have no entry/SL/TP).

7
Set resolve_after — now + timeframe_duration (15m/1h/4h/24h). This is the earliest the resolver will look at the signal.
8
Insert to signal_tracker with outcome = "PENDING".

2B. Resolving Signals (WIN / LOSS / NEUTRAL)

Runs at the top of every 5-minute cycle. Fetches all PENDING signals past their resolve_after time, gets current Coinbase prices, then applies methodology-specific resolution.

Resolution Path 1: Technical (or any signal missing SL/TP)

Uses a flat 5% threshold from entry price (which = price_at_signal for technical):

DirectionWINLOSSNEUTRAL
BUY / STRONG_BUYPrice rose ≥ 5% from entryPrice fell ≥ 5% from entry14 days elapsed, neither hit
SELL / STRONG_SELLPrice fell ≥ 5% from entryPrice rose ≥ 5% from entry14 days elapsed, neither hit

If neither threshold is hit and < 14 days have passed → stays PENDING (checked again next cycle).

Resolution Path 2: SMC / Elliott / Price Action (with SL and TP)

Holds the signal open until price hits the actual stop loss or take profit level:

DirectionWINLOSSNEUTRAL
BUY / STRONG_BUYcurrent_price ≥ take_profitcurrent_price ≤ stop_loss14 days elapsed
SELL / STRONG_SELLcurrent_price ≤ take_profitcurrent_price ≥ stop_loss14 days elapsed

If neither SL nor TP hit and < 14 days → stays PENDING.

Entry Reachability Check

For limit-order-style entries (SMC, Price Action) where entry differs from signal price by > 1%:

If the entry hasn't been "reached" by price, the signal stays PENDING regardless of SL/TP. Example: SMC gives BUY with entry at $90K (OB level) but current price is $95K. The signal waits until price drops to ~$90K, or 14 days pass (→ NEUTRAL).

entry_unreachable = (is_buy AND current > entry AND signal_price > entry)
                  OR (is_sell AND current < entry AND signal_price < entry)

if entry_unreachable AND age < 14 days:
    skip (keep PENDING)

P&L Calculation

price_change_pct = ((current_price - entry_price) / entry_price) × 100

Always from entry_price, not signal price. This is a raw directional number — positive means price went up, negative means down. For SELL signals, a negative price_change_pct is actually profitable (but the column doesn't flip the sign).

Critical resolution issues to audit:
  • Spot-check only (every 5 min) — The resolver checks the current price. If price spiked through TP and reversed through SL between two checks, the resolution depends on where price is when the check runs. Intraday wicks could hit SL/TP and reverse before the 5-min check sees it. This is especially problematic on 15m/1h timeframes.
  • Only TP1 is tracked — SMC and PA have a single TP. Elliott has 3 TPs but only TP1 is stored in the take_profit column. Resolution is based on TP1 only. If price reaches TP2/TP3 but not TP1 (shouldn't happen geometrically but worth noting).
  • price_change_pct for SELL signals is misleading — A winning SELL will have a negative price_change_pct (price fell). The avg_return_pct stat doesn't flip the sign for sells, so winning shorts drag the average return down even though they're profitable trades.
  • NEUTRAL at 14 days — Signals that never hit SL or TP become NEUTRAL. If a large fraction end up NEUTRAL, the win rate calculation (wins / resolved) could be misleadingly high because near-losses are excluded as neutral.
  • Entry reachability is one-directional — It checks if price is on the "wrong side" of the entry. But it doesn't verify price actually touched the entry level — it just checks if current price is beyond entry. If price gaps through, it's considered "reached."

Part 3 — Known Gaps & Audit Checklist

A. Signal Generation Gaps

#IssueImpactSeverity
1 SMC: Fixed 3% stop loss — Not adaptive to coin volatility or ATR Stops too tight on volatile coins, too loose on stable ones. Distorts R:R. MEDIUM
2 Elliott: Entry = current price — Always at-market, never a pullback level Entry reachability check always passes. No "limit order" behavior like SMC/PA. LOW
3 PA: 5% fallback TP — When no opposing S/R exists Arbitrary target that can inflate R:R when actual structure is absent. LOW
4 Only TP1 stored — Elliott's TP2/TP3 exist in signal_data JSONB but aren't in the top-level column Resolution only checks TP1. Can't track partial TP accuracy. LOW
5 SMC "near OB" at 3% — Generous proximity threshold Signals fire when price may never actually retrace to the OB entry level. LOW

B. Signal Accuracy Tracking Gaps

#IssueImpactSeverity
6 5-min spot-check resolution — Only checks current price, not candle highs/lows between checks Can miss SL/TP hits that occur intrabar. Signal might show WIN when SL was hit first (or vice versa), especially on shorter timeframes. HIGH
7 SELL P&L sign not flipped — price_change_pct is always (current−entry)/entry Winning SELL signals show negative returns. avg_return_pct is unreliable when mixing BUY and SELL signals. HIGH
8 NEUTRAL inflates perceived win rate — Win rate = wins/(wins+losses+neutrals) If many signals time out as NEUTRAL, the denominator grows without adding losses, making win rate look better than actual W/L ratio. MEDIUM
9 Entry reachability doesn't confirm touch — Only checks if price is "beyond" entry A gap through the entry level counts as "reached." For limit-order entries (SMC/PA), this may not reflect reality. LOW
10 No SL-first vs TP-first tracking — When both are hit in the same check window If price hits SL then bounces to TP between checks, we might record WIN (current price at TP). No way to know SL was hit first. MEDIUM
11 Technical 5% threshold is one-size-fits-all — Same threshold regardless of coin volatility 5% is easily hit on volatile coins (SOL, SUI) but hard on stable ones (BTC, ETH). Creates bias toward more volatile coins showing more wins AND losses. MEDIUM

C. Audit Checklist

Use this to systematically verify signal accuracy data is reliable:

CheckHow to VerifyWhat to Look For
SELL return signs Filter tracker for SELL WINs. Check if price_change_pct is negative. All winning SELLs should have negative %. If so, avg_return_pct is being dragged down by correct behavior. Fix: flip sign for sells in aggregation.
NEUTRAL ratio Check neutral / resolved ratio overall and per method. If > 30% NEUTRAL, the 14-day timeout is resolving too many signals. May need longer hold or different resolution logic.
Entry reachability Filter for SMC/PA signals where entry_price differs from price_at_signal by > 5%. These are limit-order entries. Check how many resolved as WIN/LOSS without price ever touching entry. May need candle-range verification.
SL/TP miss from spot-check Look at resolved signals where resolved_price is between SL and TP (not at either level). If outcome = WIN but resolved_price < TP (or LOSS but resolved_price > SL), the spot-check may have caught a temporary spike.
R:R consistency Compare R:R from frontend (client-computed) vs what quality gates checked at recording time. Should match. If TP or SL changed between recording and display, there's a data integrity issue.
Conviction predictiveness Compare HIGH vs LOW win rates per method. If HIGH doesn't meaningfully outperform LOW, the conviction logic isn't adding value and may need recalibration.
Duplicate signals Look for same symbol + method + timeframe + signal within dedup window. Should be zero after the per-TF dedup fix. If any exist, dedup logic has a bug.

D. Recommended Fixes (Priority Order)

  1. Fix SELL P&L sign HIGH
    In resolve_signals() and/or /api/signals/accuracy, flip price_change_pct for SELL signals so that a profitable short shows a positive return. This makes avg_return_pct meaningful across both directions.
  2. Add candle high/low checking to resolution HIGH
    Instead of just checking current_price, fetch the candle range (high/low) since the last check. If the candle low ≤ SL, that's a LOSS — even if current price recovered. If candle high ≥ TP, that's a WIN. For ambiguous cases (both hit), use whichever was closer to the open (approximation of "which hit first").
  3. SMC: ATR-based stop loss MEDIUM
    Replace fixed 3% stop with entry ± 1.5× ATR(14) (matching Price Action). This adapts to each coin's volatility and produces more meaningful R:R.
  4. Report W/L ratio alongside win rate MEDIUM
    Add wins / (wins + losses) as a separate metric that excludes NEUTRALs. Show both to understand true predictive accuracy vs the timeout-adjusted rate.
  5. ATR-scaled technical threshold MEDIUM
    Replace fixed 5% technical resolution threshold with 3 × ATR(14) / price × 100. This normalizes resolution to each coin's volatility.
